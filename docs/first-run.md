# 首次开机引导卡

用途：新项目第一次复制本启动包后，帮助人和 AI 一起补齐项目身份，避免在项目还没说清楚时进入技术路线、backlog、foundation 或 coding。

本文件只是一张引导卡，不是项目身份的存放处。项目身份最终必须写入：

- `AGENTS.md` / `CLAUDE.md` 的“本项目的快速介绍了解”（1-3 行摘要）
- `docs/prd.md`（完整项目身份、产品定位、目标用户、范围边界和验收标准）

## 触发条件

出现以下任一情况时，必须先进入“首次开机”流程：

- “本项目的快速介绍了解”仍是占位。
- `docs/prd.md` 的产品名称为空。
- `docs/prd.md` 的产品定位为空。
- 用户只说“我要开一个新项目 / 初始化项目 / 用这个启动包”，但没有给出清晰项目身份。

触发后不得进入：

- 技术路线
- backlog 拆分
- foundation
- coding

## 人和 AI 的分工

- 人负责判断：项目到底要解决什么问题、服务谁、哪些边界不能碰。
- AI 负责澄清：把模糊想法整理成可写入 `AGENTS.md` / `CLAUDE.md` 和 `docs/prd.md` 的清晰表述。
- AI 可以提出默认猜测，但必须标注“待确认”。
- 用户确认前，不要把猜测写成已确定事实。

## 轻量启动原则

首次开机要尽量轻，不要把用户拖进完整 PRD 填表。

- 第一轮最多问 3 个问题，除非用户主动展开。
- 信息达到 60% 清晰时，先生成“可用项目身份 v0”。
- AI 先给默认草案，再让用户修正；不要要求用户一次性写完整 PRD。
- 第一轮只强制落 3 个字段：项目名、一句话定位、不做什么。
- 进入技术路线、开发级 PRD、foundation 或 coding 前，再补完整身份字段。

## 三分钟补齐问题

AI 第一次接手空白项目时，先问这 3 个问题：

1. 这个项目叫什么？如果还没名字，可以先用临时名。
2. 一句话说，它是给谁用、解决什么问题的？
3. 有哪些明确不做、不能碰、不能误解的边界？

用户可以直接粗略回答。AI 的任务是把回答整理成可用版本，而不是要求用户一次性写完整 PRD。

如果用户已经回答得比较清楚，AI 可以不继续追问，直接给“项目身份 v0”草案让用户确认。

必要时再补问 2 个问题：

4. 你最希望它第一阶段做到什么程度，才算有用？
5. 你当前想推进到哪一步：机会判断、技术路线、PRD、foundation，还是某个具体功能？

## 最小落地结果

轻量首次开机完成后，至少要产出三件事：

1. `AGENTS.md` / `CLAUDE.md` 的“本项目的快速介绍了解”已改写为当前项目摘要。
2. `docs/prd.md` 至少补齐：
   - 产品名称
   - 产品定位：一句话定位
   - 不做什么
3. `docs/iteration-log.md` 的“当前状态摘要”已写明：
   - 当前阶段
   - 最近完成
   - 当前阻塞
   - 下一步唯一建议

轻量首次开机只表示“项目身份 v0 已可识别”，不代表已经可以进入技术路线、开发级 PRD、foundation 或 coding。

## 完整身份补齐条件

如果用户要继续进入技术路线、开发级 PRD、foundation 或 coding，必须在 `docs/prd.md` 继续补齐：

- selected project direction
- 状态：概念 PRD / 开发级 PRD
- 产品边界
- 目标用户
- 核心问题
- MVP 范围或明确待定说明

这些字段未补齐时，只能继续澄清、机会判断或概念 PRD，不得进入实际开发。

## 输出格式

首次开机结束时，AI 应给用户一个简短确认：

```text
已完成首次开机：

- 项目身份摘要：____
- 项目身份 v0 已写入：docs/prd.md
- 当前阶段：____
- 下一步唯一建议：执行 ____

本轮没有进入技术路线、backlog、foundation 或 coding。
```

## 防线

- 不要因为用户只有一个模糊想法，就直接写完整开发级 PRD。
- 不要因为项目名已确定，就默认可以进入 coding。
- 不要把 `docs/first-run.md` 当成新的项目身份文档；它只负责引导第一次补齐身份。
- 不要为了完整流程追问过多；先给项目身份 v0，再逐步补齐。
- 如果用户只想先讨论方案，不要修改文件。
